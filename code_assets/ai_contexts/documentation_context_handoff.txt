DOCUMENTATION CONTEXT HANDOFF
==============================

Purpose
-------
This handoff is meant to be pasted into a future conversation so another model can quickly recover the documentation-review context for the TIFF viewer / camera-calibration app. The goal is to let the future model do the same kind of work that was done in this conversation:
1. critically review the docs against the actual source code,
2. identify mismatches, missing behavior details, weak terminology, and insufficient operator guidance,
3. then generate replacement documentation files that match current code behavior.

This handoff is intentionally long and explicit. It captures both the user's expectations and the review methodology that worked well.

----------------------------------------------------------------------
1. USER EXPECTATIONS AND DOCUMENTATION PHILOSOPHY
----------------------------------------------------------------------

The user wants documentation that is:
- professional-grade,
- technically meaningful,
- concise enough to remain usable,
- but complete enough that an engineer can operate and maintain the app without guesswork.

The user explicitly does NOT want:
- fluffy language,
- casual phrasing,
- beginner-style simplifications that sound non-technical,
- vague statements like "it kind of helps" or "partially factors out" when more precise wording is possible.

The preferred tone is:
- engineering-oriented,
- explicit about assumptions and limits,
- operationally useful,
- mathematically honest,
- behavior-level rather than marketing-level.

Examples of wording preferences:
- Prefer "normalizes the result by integration duration, allowing first-order comparison across images acquired with different exposure settings"
  over
  "partially factors out exposure duration".
- Prefer exact behavioral descriptions over generic claims.
- Prefer "analysis plugin" over "analysis profile" unless true saved profiles exist.
- Prefer "defaults are inherited baseline values; overrides replace targeted values; blank fields do not erase inherited values"
  over vague datacard descriptions.

The user may ask for:
- single-page replacements,
- whole-folder replacements,
- complete docs-folder substitutes,
- top-level maintenance file updates such as mkdocs.yml, README.md, scripts/docs/check.py or others.

The user also wants deliverables packaged as files, not just pasted chat content.
Ideal workflow:
- analyze the code,
- analyze the existing markdown docs,
- provide findings and critique,
- then produce drop-in replacement markdown files,
- then provide zip bundles containing the replacement docs tree when requested. If maintenance files were updated,
  provide them in the zip at their original position relative to the root directory.

----------------------------------------------------------------------
2. REVIEW METHOD THAT WORKED WELL
----------------------------------------------------------------------

The most successful review approach in the earlier conversation was this:

A. Read the source code first enough to verify actual behavior.
B. Read the markdown docs against the implementation.
C. Identify:
   - terminology mismatches,
   - behavioral mismatches,
   - missing exact rules,
   - edge cases,
   - places where docs stop too early and do not tell the user how to choose settings,
   - places where docs describe UI but not interpretation limits.

D. Give an opinionated but precise review:
   - what is already strong,
   - what is structurally good,
   - what is weak,
   - what is risky,
   - what must be fixed before polish,
   - what needs precision versus full rewrite.

E. Then rewrite page-by-page.

The user responded well when the review was framed like:
- "The structure is good, but the docs behave more like a strong draft than a finished operator manual."
- "The problem is not architecture anymore; it is precision, interpretation guidance, and consistency."
- "The main gap is that the docs describe the app at a feature level instead of a behavior level."

That framing is useful and should be reused when appropriate.

----------------------------------------------------------------------
3. IMPORTANT HISTORICAL FINDINGS FROM THE EARLIER REVIEW
----------------------------------------------------------------------

These were the major findings that mattered in the earlier pass. A future model should re-check them against the new codebase rather than blindly assume they still hold.

3.1 Datacard defaults vs overrides was previously the biggest concern
--------------------------------------------------------------------
A critical early finding was that the original docs said:
- defaults apply acquisition-wide,
- frame-specific overrides are only needed where defaults are not enough.

But the then-current implementation appeared to resolve metadata in a way that did NOT always inherit defaults automatically for frames lacking overrides.

That was flagged as the highest-priority doc/code mismatch.

Later, the user provided a patched metadata.py, and the concern was re-checked. After the patch, the interpretation was:

- defaults are inherited baseline metadata,
- overrides are applied as patches on top of the inherited base context,
- if no override matches, defaults still remain available,
- blank values mean "do not set anything here", not "erase inherited value",
- explicit clearing via null is not part of normal supported semantics.

This became the accepted semantic model for the app:
- defaults = inherited baseline,
- overrides = concrete replacements for targeted frames/ranges,
- blank fields = leave inherited value alone,
- no supported inheritance-plus-clearing model in normal wizard usage.

A future model MUST re-check the new source code to confirm whether this still holds.

3.2 "Analysis Profile" was misleading terminology
-------------------------------------------------
A key documentation issue was that "analysis profile" was being used in the docs, but in code the selector was actually choosing between loaded analysis plugins, not saved user profiles.

Recommended terminology became:
- analysis plugin
- active analysis plugin
- plugin selector

A future model should verify whether the app now truly supports saved analysis profiles. If not, keep the plugin wording.

3.3 Normalization affected more than display
--------------------------------------------
A critical nuance found in code:
- normalization did not rewrite TIFF source data,
- but it DID affect displayed intensity-derived values,
- and it also propagated into analysis-context values such as mean/std/sem and dn_per_ms-related fields.

Therefore the preferred documentation statement became:
"Normalization does not rewrite the source TIFF pixels, but it does affect displayed measurement values and the intensity-derived values passed to analysis plugins."

A future model should verify the current behavior in:
- measurement-stage context building,
- analysis-context construction,
- plugin input preparation.

3.4 Grouping rules were more exact than the docs implied
--------------------------------------------------------
Earlier docs described grouping in a more open-ended way, but the code implemented a fixed menu with exact-token grouping behavior.

Previously observed fixed options included:
- None
- Parent Folder
- Grandparent Folder
- Iris Position
- Exposure [ms]

And earlier exact grouping behavior included:
- grouping by exact string token,
- empty or missing values becoming group 0,
- grouping None making everything group 1,
- non-empty values stringified, sorted, and assigned IDs from 1.

A future model must re-check whether grouping options have expanded in the new app version.

3.5 Background correction needed exact operational wording
----------------------------------------------------------
Earlier code inspection found that folder-library background matching behaved more precisely than the docs said. Previously observed behavior:
- exposure-keyed matching,
- exact rounded exposure matching such as round(exposure_ms, 9),
- exact shape match required,
- multiple references per exposure merged via median stack,
- no nearest-neighbor matching,
- no interpolation,
- unmatched images falling back to raw image values.

This level of exactness mattered to the user because the app is used for calibration work. A future model should verify the current background matching rules in code and document them exactly.

3.6 ROI file contract was important and underdocumented
-------------------------------------------------------
Earlier code review identified that ROI persistence needed contract-style documentation. Earlier observed contract:
- JSON fields: x0, y0, x1, y1, image_width, image_height
- full-resolution image coordinates
- top-left origin
- applied as image[y0:y1, x0:x1]
- x1 and y1 exclusive
- out-of-bounds ROI files rejected on load

This belongs in the docs because users may save, share, or externally generate ROI files.

3.7 Measure page needed more decision guidance, not more formulas
-----------------------------------------------------------------
The user appreciated when it was explicitly stated that the formulas were already mostly fine, but the docs still needed to explain:
- when Top-K is preferable to ROI,
- how to choose K,
- how threshold affects saturation counts,
- when background subtraction improves comparability,
- when clipping negatives biases the result,
- when DN/ms is useful,
- when DN/ms is not physically meaningful enough to compare directly.

This remains a central documentation principle for future rewrites.

3.8 Intensity Trend Explorer was mathematically strong but still needed operational emphasis
--------------------------------------------------------------------------------------------
The plugin docs already had strong math, but needed clearer explanations of what was ACTUALLY plotted in each mode.

Earlier examples that needed clarification:
- special-case Iris Position + Gain behavior,
- peak intensity normalization wording,
- what curves are built from,
- when aggregation occurs,
- what error bars actually reflect,
- that a good-looking plot can still be wrong if upstream metadata/measurement choices were wrong.

A future model should preserve the strong math but split the page into:
- practical operator interpretation,
- mathematical appendix.

3.9 Troubleshooting needed result-quality sections, not only UI issues
----------------------------------------------------------------------
Earlier troubleshooting docs were too focused on:
- no TIFF files found,
- missing metadata,
- Analyze tab missing,
- datacard wizard issues,
- help link/browser issues.

The user agreed that troubleshooting should also cover:
- DN/ms blank or missing,
- ROI metrics NaN,
- suspiciously flat or discontinuous gain curves,
- wrong grouping,
- background library partially unmatched,
- plot disagreeing with preview/table,
- raw vs normalized confusion,
- datacard values not appearing as expected.

This "result-quality troubleshooting" concept should be preserved.

3.10 Some practical features had been undocumented
--------------------------------------------------
One earlier example was exporting the image metrics table via File -> Export.
A future model should actively look for user-facing features that are easy to miss in docs:
- export commands,
- menu actions,
- plugin-specific tools,
- help or reload behaviors,
- config utilities,
- new data tools added in newer versions.

4. DOCUMENTATION DESIGN PRINCIPLES THAT EMERGED
----------------------------------------------------------------------

These became the effective style guide for the rewritten docs.

4.1 Structure
-------------
The docs architecture was considered fundamentally sound when it used:
- User Guide
- Reference
- Developer Guide
- Troubleshooting

Within the User Guide, the workflow split:
- Data -> Measure -> Analyze
was considered the correct backbone.

The docs usually benefited from:
- adding a "Concepts and Limits" page early,
- keeping workflow pages consistent,
- separating operator guidance from schema/reference material.

4.2 Page responsibilities
-------------------------
User Guide pages should answer:
- what the user does,
- how to choose settings,
- what the outputs mean,
- what limits and assumptions apply,
- what failures or warnings matter operationally.

Reference pages should answer:
- exact schema keys,
- file locations,
- config contracts,
- shortcut scope,
- manifest fields,
- mapping definitions,
- payload validity expectations.

Developer Guide pages should answer:
- lifecycle,
- state ownership,
- subsystem boundaries,
- extension contracts,
- invariants,
- packaging/build realities,
- what to change for a given type of modification.

Troubleshooting should answer:
- where the failure likely originates,
- what to verify next,
- whether the problem is intake, metadata, measure, analysis, plugin, or docs packaging.

4.3 Behavior-level documentation
--------------------------------
A repeated theme: documentation should describe the app at a behavior level, not merely at a feature level.

Needed categories:
- exact rules,
- assumptions,
- limits,
- edge cases,
- interpretation guidance,
- failure modes.

4.4 Engineering tone
--------------------
Preferred characteristics:
- precise,
- technical,
- compact but not cryptic,
- no marketing tone,
- no patronizing simplifications,
- no overstatement of certainty.

9. WHAT A FUTURE MODEL SHOULD DO WHEN REPEATING THIS TASK
----------------------------------------------------------------------

When given a new source-code zip and existing docs, the future model should do the following:

Step 1: inspect the repo structure
- identify docs tree structure,
- identify plugin folders,
- identify main_window modules,
- identify metadata and datacard code,
- identify measurement pipeline,
- identify analysis plugins,
- identify config/reference assets,
- identify scripts/docs build/check pipeline.

Step 2: inspect markdown docs
- compare current docs tree against actual repo features,
- identify weak/obsolete/misaligned pages,
- detect terminology drift,
- detect missing pages for new features/plugins.

Step 3: perform a critique before rewriting
The critique should cover:
- what is already good,
- what needs only precision,
- what needs real expansion,
- what is risky,
- what is missing entirely,
- what should remain reference-only vs user-guide vs dev-guide.

Step 4: verify critical implementation contracts
At minimum, re-check:
- datacard metadata resolution semantics,
- normalization propagation into analysis,
- exact grouping rules,
- background matching logic,
- ROI save/load contract,
- analysis-plugin selection model,
- plugin discovery/manifest behavior,
- config file ownership/location/creation behavior,
- any new eBUS or related tool contracts.

Step 5: rewrite docs folder in one coherent pass
Preferred output:
- either page-by-page drop-ins,
- or a full docs-folder substitute zip.

Step 6: verify docs maintenance files if needed
Check:
- mkdocs.yml
- scripts/docs/build.py
- scripts/docs/check.py
- docs_layout.md
Only update them when the docs tree/build contract truly changed.

10. IMPORTANT CAUTIONS FOR A FUTURE MODEL
----------------------------------------------------------------------

Do NOT:
- assume earlier code behavior is still current,
- assume earlier docs bundles are still valid,
- assume path layout is unchanged,
- assume build/check scripts need updates without verifying,
- over-document beginner material at the expense of precise behavior,
- conflate workflow docs with reference docs,
- use "analysis profile" unless the code really supports profiles,
- describe normalization as purely cosmetic unless code confirms that,
- describe datacard defaults as inherited unless current metadata resolution still does that.

Do:
- re-check implementation every time,
- prefer exact language,
- document engineering limits explicitly,
- distinguish raw/normalized/corrected contexts carefully,
- treat developer guide as a maintenance manual, not a summary brochure,
- preserve strong mathematical sections but front-load practical interpretation.
