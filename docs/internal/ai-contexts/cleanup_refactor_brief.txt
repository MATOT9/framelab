Hard rule:
- Never inspect, use, or consider the `archived` folder unless I explicitly ask.

Goal:
- Do a high-value cleanup/refactor pass on the target module(s)/area(s).
- Focus on maintainability, removal of duplication, removal of dead-end/leftover code, simplification of bloated modules/classes, and small optimizations that clearly improve both code quality and performance.
- Do not chase perfection or split files for no reason. Use common sense. Create folders/packages when it improves structure and avoids bloating the top-level package.
- Preserve useful extension hooks and plugin architecture. Remove leftovers, not future-facing hooks.

Important constraints:
- Keep behavior unchanged unless I explicitly ask for behavior changes.
- Respect packaging/bundling concerns: avoid adding unnecessary third-party dependencies.
- Prefer fewer, cleaner modules over many tiny files.
- PEP 8 everywhere.
- Use NumPy-style docstrings where applicable.
- Keep imports minimal and dependency boundaries clean.
- If something is duplicated in multiple places, centralize it only if the shared abstraction is genuinely reusable and not business-logic overreach.
- Treat any code whose stated purpose is to preserve compatibility with older behavior, interfaces, formats, or integrations as off-limits unless I explicitly request changes to it.

What “do the same” means in this repo:
1. Identify the obvious high-value cleanup candidates first.
2. Consolidate low-level shared helpers that are duplicated across modules.
3. Split monolithic modules/classes by coherent responsibility.
4. Group related modules in subpackages when that improves maintainability.
5. Remove dead code, stale bridges, obsolete compatibility layers, and duplicate constants/mappings.
6. Keep plugin discovery/registration working after refactors.
7. Verify with compile/import checks after each meaningful restructuring step.
8. Stop when there are no more obvious high-value cleanups in that specific area.

Current architectural preferences for future refactors:
- If a large module represents one coherent page or feature, keep it intact unless there is still clear internal cross-cutting bloat.
- If a module mixes unrelated responsibilities, split it by responsibility.
- For plugins, package-per-plugin is acceptable and preferred once the plugin becomes large enough.
- Use internal helper modules with leading underscore inside plugin packages for non-entrypoint code.
- Keep the public/plugin entrypoint shallow and obvious.

Cleanup principles to apply:
- Remove duplicate constants if there is already a JSON/config-backed source of truth.
- Remove dead state/unused widgets/unused parameters when they are clearly leftovers.
- Tighten typing when there is a more specific local type available.
- Avoid stale bridge modules once the real code has been relocated.
- Avoid eager heavyweight imports at package import time when lazy boundaries are better.

Repo-specific functional constraints already established:
- Local config should live in the app folder (`config/`), not user home config dirs, because the app should be shareable.
- Mapping-driven metadata/label/tooltip configuration should come from JSON where appropriate.
- If a JSON mapping entry has an empty or missing tooltip, no tooltip should be shown.
- Only group labels should use bold styling across the app.
- Tooltip styling should be consistent app-wide.
- Platform-specific styling fixes should prefer explicit widget styling over inherited native theme behavior.

- The `iris_gain` plugin remains a package, not a single monolithic file.
